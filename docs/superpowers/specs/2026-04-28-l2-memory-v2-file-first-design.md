# L2 Memory v2 File-First Design

## Goal

Move Vulture memory from a DB-first product feature to an OpenClaw-style file-first operator feature.

The long-term memory source of truth becomes Markdown files in the agent workspace. SQLite remains the local index/cache used for retrieval, embeddings, UI state, and migration bookkeeping. Agents interact with memory through OpenAI Agents SDK tools, with prompt injection reduced to a compact memory summary and retrieval guidance.

This design builds on `2026-04-28-l2-memory-vector-store-design.md`. The existing `memories` table is treated as a migration source and compatibility layer, not the final canonical memory store.

## Context

Vulture is a desktop developer/operator tool, not a consumer app that must hide configuration complexity. Memory should therefore be transparent, manually editable, easy to back up, and friendly to agent workflows.

OpenClaw's strongest memory ideas to absorb:

- Markdown memory files as durable operator-owned state.
- SQLite/FTS/vector indexing as derived retrieval state.
- `memory_search` and `memory_get` tools for progressive disclosure.
- Candidate memory staging before promotion into long-term memory.
- Append-only guarded writes for agent-managed memory updates.
- Later support for session snapshots, compaction flushes, and background consolidation.

## Scope

In scope for the first implementation plan:

- Define a file-first memory root under each agent workspace.
- Migrate current DB memories into `MEMORY.md`.
- Add a memory file indexer backed by SQLite.
- Add `memory_search`, `memory_get`, and `memory_append` core tools.
- Change run context from top-k `<memories>` injection to memory summary plus tool guidance.
- Add suggestion storage for pending memory candidates.
- Add Settings memory UI for file/index/suggestion management.
- Preserve existing manual memory API behavior through compatibility routes during migration.

Out of scope for the first implementation plan:

- Full OpenClaw dreaming/background promotion.
- Active-memory sub-agent before every run.
- Compaction-triggered memory flush.
- Cross-agent shared memory.
- Cloud sync or hosted vector stores.
- Deleting user-authored memory file content from UI without an explicit file edit.

## Memory Layout

Each agent already has a workspace under `~/.vuture/workspace/<agent-slug>`. Memory files live inside that workspace:

```text
~/.vuture/workspace/<agent-slug>/
  MEMORY.md
  memory/
    2026-04-28.md
    sessions/
    suggestions/
```

`MEMORY.md` is the stable long-term memory file. It should contain durable project facts, user preferences, conventions, and standing instructions.

`memory/YYYY-MM-DD.md` files are daily or session-level notes. They can contain run summaries, temporary observations, and context that may later be promoted to `MEMORY.md`.

`memory/sessions/` is reserved for later session snapshots.

`memory/suggestions/` stores pending candidate memories generated by the system. Suggestions are not authoritative until accepted.

## SQLite Index

SQLite stores derived state for fast retrieval and UI inspection.

Add or extend tables for:

- `memory_files`
  - `id`
  - `agent_id`
  - `path`
  - `mtime_ms`
  - `content_hash`
  - `indexed_at`
  - `status`
  - `error_message`

- `memory_chunks`
  - `id`
  - `agent_id`
  - `file_id`
  - `path`
  - `heading`
  - `content`
  - `keywords_json`
  - `embedding_json`
  - `start_line`
  - `end_line`
  - `content_hash`
  - `updated_at`

- `memory_suggestions`
  - `id`
  - `agent_id`
  - `run_id`
  - `conversation_id`
  - `content`
  - `reason`
  - `target_path`
  - `status`: `pending`, `accepted`, or `dismissed`
  - `created_at`
  - `updated_at`

The existing `memories` table remains for compatibility until migration is complete. New canonical writes should go to files and then be indexed.

## Indexer

The indexer scans:

- `MEMORY.md`
- `memory/**/*.md`

It ignores:

- `memory/suggestions/**/*.md` for authoritative search, unless a suggestion is accepted.
- Hidden directories other than explicitly supported internal memory paths.
- Files outside the agent workspace.

Indexing steps:

1. Resolve the agent memory root.
2. Ensure `MEMORY.md` and `memory/` exist.
3. Read candidate Markdown files.
4. Compute file hash and compare mtime/hash with `memory_files`.
5. Chunk Markdown by headings and size.
6. Generate keywords with the existing CJK-aware tokenizer.
7. Generate embeddings when the embedding provider is configured.
8. Replace chunks for changed files transactionally.

Retrieval must work without embeddings. Keyword fallback remains deterministic and testable.

## Tools

Memory tools are normal gateway core tools adapted to OpenAI Agents SDK function tools.

### `memory_search`

Input:

```json
{
  "query": "string",
  "limit": 5
}
```

Output:

```json
{
  "items": [
    {
      "id": "memchunk-...",
      "path": "MEMORY.md",
      "heading": "Project",
      "score": 0.82,
      "source": "embedding",
      "snippet": "Project codename is Vulture."
    }
  ]
}
```

`memory_search` is safe and does not require approval. It only searches the active agent's indexed memory.

### `memory_get`

Input:

```json
{
  "id": "memchunk-...",
  "path": null
}
```

or:

```json
{
  "id": null,
  "path": "MEMORY.md"
}
```

Output includes file path, heading, line range, and content. Reading by path is restricted to the agent memory root.

`memory_get` is safe when the resolved path is inside the memory root.

### `memory_append`

Input:

```json
{
  "path": "MEMORY.md",
  "content": "- User prefers concise Chinese answers."
}
```

Allowed targets:

- `MEMORY.md`
- `memory/YYYY-MM-DD.md`

`memory_append` is append-only and requires approval. It must reject path traversal, absolute paths, binary files, and writes outside the agent workspace memory root.

After a successful append, the gateway reindexes the changed file.

## Prompt Strategy

The current top-k `<memories>` block is replaced with progressive disclosure:

1. Run context includes a short memory summary and tool guidance.
2. The agent is told that durable memory is available through `memory_search` and `memory_get`.
3. The agent should search memory when the user asks about prior preferences, project facts, historical decisions, or durable context.
4. The agent should use `memory_append` only for stable durable facts and only when the user asked to remember something or when the app creates an approved suggestion.

Initial prompt shape:

```text
Memory is available for this agent.

Use memory_search when durable user/project context may help.
Use memory_get to inspect full memory entries before relying on details.
Use memory_append only for approved durable memory updates.

Memory summary:
- Project codename is Vulture.
- User prefers direct Chinese engineering guidance.
```

The summary is derived from `MEMORY.md` and capped to a small budget. It is not a replacement for tool-based lookup.

## Automatic Suggestions

After a successful assistant run, Vulture can run a no-tools extraction pass to propose durable memory candidates.

Extraction input:

- User message.
- Assistant final output.
- Tool summaries, not full large outputs.
- Existing memory summary.

Extractor output:

```json
{
  "suggestions": [
    {
      "content": "Project codename is Vulture.",
      "reason": "The user asked the agent to remember the project codename.",
      "targetPath": "MEMORY.md"
    }
  ]
}
```

Suggestions are stored as pending. They are not injected into future runs until accepted.

Acceptance appends the content to `targetPath`, marks the suggestion accepted, and reindexes the file. Dismissal marks it dismissed.

The first implementation should make automatic extraction an explicit opt-in operator feature. Suggestion storage and API can exist, but ordinary user-facing settings should not expose pending suggestion review until Vulture has a dedicated advanced/operator mode.

## UI

The regular Settings Memory section should stay understandable as file-backed memory management:

- Show active agent and memory root.
- Show whether `MEMORY.md` exists.
- Show index status, indexed file count, indexed chunk count, and last indexed time.
- Provide a reindex button.
- Show indexed memory chunks with path, heading, snippet, and timestamp.
- Keep manual add behavior by appending to `MEMORY.md` instead of writing DB-only memory.

Pending suggestions with accept/dismiss actions belong in a future advanced/operator console, not the ordinary user-facing Settings page.

## Migration

On gateway startup or first memory access:

1. Ensure each agent memory root exists.
2. If `MEMORY.md` does not exist, create it with a heading.
3. If legacy DB memories exist for the agent and migration has not completed, append them to `MEMORY.md` under a `Migrated Memories` heading.
4. Mark migration complete so repeated startup does not duplicate entries.
5. Reindex memory files.

Legacy memory APIs:

- `GET /v1/agents/:agentId/memories` returns indexed authoritative chunks or a compatibility view.
- `POST /v1/agents/:agentId/memories` appends to `MEMORY.md` and reindexes.
- `DELETE /v1/agents/:agentId/memories/:memoryId` can delete legacy DB rows during compatibility, but must not silently edit Markdown chunks. For file-backed chunks, the route should return a clear unsupported/manual-edit response unless a dedicated file edit flow is implemented.

## Configuration

Agent-level configuration should eventually include:

- memory enabled/disabled.
- memory tools enabled/disabled.
- memory root override.
- auto suggestion extraction enabled/disabled.
- allowed append targets.
- embedding provider behavior.

For the first implementation, defaults are enough:

- memory enabled for all agents.
- tools available through the existing tool allowlist when selected.
- memory root is the agent workspace.
- auto suggestion extraction is disabled unless `VULTURE_MEMORY_SUGGESTIONS=1`.
- append targets are restricted to `MEMORY.md` and today's daily note.

## Security and Trust Boundary

Memory files are operator-owned local state. If a person edits them, Vulture treats those edits as trusted agent context.

Tool access still needs strict path controls:

- No path traversal.
- No absolute paths in memory tool inputs.
- No writes outside the active agent workspace.
- Append-only writes for `memory_append`.
- Approval required for memory writes.
- Search/get only return memory files for the active agent.

Prompt text should describe memory as durable context, not as higher-priority system instructions. Existing system/developer instructions still outrank memory content.

## Error Handling

- Indexing failure marks the file status and keeps other indexed files usable.
- Embedding failure falls back to keyword indexing/search.
- Search with no index returns an empty result and a clear status.
- Migration failure logs the error and leaves existing DB memory untouched.
- Suggestion extraction failure does not affect the user-facing assistant run.
- Append failure returns a tool error and does not mark suggestions accepted.

## Testing

Gateway tests:

- Migration writes legacy DB memories to `MEMORY.md` exactly once.
- Indexer creates file and chunk rows for `MEMORY.md`.
- Indexer updates chunks when the file hash changes.
- Keyword search retrieves CJK and English memory chunks without embeddings.
- Embedding search ranks chunks when vectors are available.
- `memory_search` only returns active agent memory.
- `memory_get` rejects paths outside memory root.
- `memory_append` requires approval and rejects unsafe paths.
- Accepted suggestion appends to `MEMORY.md` and reindexes.
- Run context includes memory summary and tool guidance, not the old full top-k block.

Desktop UI tests:

- Memory page displays memory root and index status.
- Manual add appends to `MEMORY.md` through the compatibility API.
- Reindex refreshes visible chunks.
- Ordinary Settings does not expose pending suggestions.

Regression tests:

- Existing skill context still combines with memory context.
- Existing run recovery and tool streaming behavior remains unchanged.
- Gateway and desktop typechecks pass.

## Acceptance Criteria

- A new agent workspace contains `MEMORY.md` and `memory/` after memory initialization.
- Existing DB memories migrate into `MEMORY.md` without duplication.
- Asking about a known memory can trigger `memory_search` and `memory_get`.
- Manual memory add persists as Markdown and appears in indexed results.
- `memory_append` cannot write outside the agent memory root.
- The Memory settings page shows file-backed memory state without exposing pending suggestion review to ordinary users.
- Runs remain functional when embeddings are unavailable.
- Existing tests pass after updating expected memory prompt behavior.

## Implementation Phases

Phase 1: file-first foundation and compatibility.

- Memory root resolver.
- Legacy DB migration to `MEMORY.md`.
- Markdown indexer.
- Compatibility route updates.
- Memory page root/index status.

Phase 2: SDK tools and prompt strategy.

- `memory_search`.
- `memory_get`.
- `memory_append`.
- Tool allowlist integration.
- Run prompt summary/guidance change.

Phase 3: suggestions.

- Suggestion table/routes.
- Async extractor service.
- Settings UI accept/dismiss.
- Append and reindex on accept.

Phase 4: later OpenClaw-style upgrades.

- Compaction pre-flush.
- Session snapshots.
- Active-memory recall sub-agent.
- Background consolidation/dreaming.
